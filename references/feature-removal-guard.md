# 功能删减保护（Feature Removal Guard）

目标：把“为了更简单而减少既有功能”的路径从默认可做，改成默认禁止且必须先问。

原则：
- 默认保留现有功能。
- 未经用户明确批准，禁止通过删除、隐藏、禁用、短路、降级既有功能来完成任务。
- “更简单”“更小 diff”“更易维护”“先跑通”不构成功能删减授权。

---

## 默认原则

- **preserve_by_default**：默认保留现有功能与既有契约。
- **ask_before_reduce**：一旦疑似会减少功能，必须先停下询问用户。
- **no_implicit_approval**：不得把“简化实现/最小改动面/顺手优化”解释为删功能授权。
- **protect_public_surface**：凡是用户可见表面、公开契约、默认能力，默认都按“受保护功能”处理。

---

## 用户可见 / 公开表面识别

命中以下任一表面时，若行为会减少/隐藏/禁用/降级，默认按“功能删减”处理：

- **用户可见表面（user_visible_surface）**
  - 页面、按钮、菜单、路由、对话框、表单字段、提示入口、结果面板
  - CLI 命令、子命令、flag、输出项、生成文件/报表入口
  - 工作流中的默认步骤、原本可直接使用的交互入口
- **公开契约表面（public_contract_surface）**
  - 公开 API、事件、导出符号、配置键、环境变量、文件格式、退出码、脚本入口
  - 文档中对外承诺的行为、兼容路径、使用方式
- **默认能力表面（default_capability_surface）**
  - 原本默认开启的能力被改成默认关闭
  - 原本支持的输入/输出场景、兼容分支、边界行为被收缩

证据优先：
- 只要代码、测试、文档、模板、输出格式、脚本入口中仍能证明该表面对外存在，就默认视为受保护功能。
- “已经很少人用 / 代码不好维护 / 先删再说” 不构成失去保护资格的证据。

---

## 视为功能删减（命中任一即触发）

- 删除或隐藏命中的用户可见表面 / 公开契约表面 / 默认能力表面。
- 将原本会执行的逻辑改为不执行、提前返回、永久关闭、默认禁用。
- 删除承载某项受保护能力的模块、文件、导出符号、脚本入口。
- 通过减少支持场景、兼容路径、边界行为来换取实现更简单。
- 将“默认开启”改为“默认关闭”，或把现有能力改成需要额外开关/额外步骤才能恢复。
- 删除仍对应有效受保护能力的测试、文档入口或公开说明。

---

## 不视为功能删减

- 删除死代码、未使用 import、无用注释、未使用 helper、临时调试代码。
- 删除经过代码事实证明永远不可达的分支。
- 不改变外部行为的重命名、搬移、内部重构。
- 删除与成功标准无关、且不承载用户可见行为或公开契约的重复抽象。
- 清理仅在内部使用、且删除后不会改变默认行为/公开入口/兼容矩阵的胶水代码。

### 内部-only 白名单（命中全部才可按内部清理处理）

- 该对象**不**属于“用户可见 / 公开表面识别”中的任一类。
- 删除后外部行为、默认行为、公开契约、兼容路径均不变。
- 不需要用户新增操作来“恢复原能力”。
- 不会让既有测试/文档/模板/脚本入口失真。

---

## 歧义处理

- 无法明确证明“这是删冗余，不是删功能”时，默认按“功能删减”处理。
- 一旦触发歧义处理，必须先询问用户，禁止继续沿删减路径实施。

---

## 判定顺序

按以下顺序判定，避免把“内部清理”误伤成“删功能”：

1. **先识别表面**：是否命中“用户可见 / 公开表面识别”中的任一类？
   - 命中：默认受保护，继续第 2 步。
   - 未命中：继续第 3 步。
2. **再看行为变化**：是否会删除、隐藏、禁用、短路、降级、收缩兼容或改变默认行为？
   - 是：判定为功能删减，必须先问。
   - 否：只有在外部行为与公开契约完全不变时，才可转到内部清理判断。
3. **检查内部-only 白名单**：是否同时满足全部白名单条件？
   - 是：不视为功能删减。
   - 否：按功能删减处理。
4. **替代实现不等于未删减**：即使存在替代路径，只要用户可见入口/公开契约/默认能力发生减少，仍属于功能删减，必须先问。

---

## 审批令牌

- `feature_removal_risk: clear|suspected|approved`
- `feature_removal_approved: yes|no`
- `approved_scope:`
- `approved_target:`
- `approved_reason:`
- `replacement_behavior:`
- `source: [SRC:USER]`

规则：
- `feature_removal_risk: clear`：当前无删功能风险，可按正常路径推进
- `feature_removal_risk: suspected`：当前存在删功能风险，且尚未完成用户批准；hooks / 主流程应优先把它当作结构化强信号
- `feature_removal_risk: approved`：用户已明确批准当前删功能路径；仍需保留批准范围、目标与替代行为
- 若 hooks / 守卫是通过 prompt 启发式首次识别到删功能高风险，应先把状态**归一化写成** `feature_removal_risk: suspected`，再阻断 / 等待审批，避免后续续作只能依赖自然语言描述
- 默认值必须为 `feature_removal_approved: no`
- 只有当来源为 `[SRC:USER]` 的明确批准存在时，才允许改为 `yes`
- 若用户拒绝删减，必须回到“保留现有功能”的实现路径或停止任务

---

## 命中后的固定动作

1. 立即停止实施，不得继续修改相关文件。
2. 在 `task.md##上下文快照` 写入：
   - 功能删减目标
   - 原因
   - 审批状态
   - 待用户输入
   - 下一步唯一动作
3. 输出“功能删减确认”交互模板。
4. 未获批准前，禁止把该路径当作默认实现方案继续推进。
