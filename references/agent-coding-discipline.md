# Agent 编码执行纪律（Agent Coding Discipline）

目标：把通用 “CLAUDE.md 12 rules” 吸收成 helloagents 的轻量执行纪律；不新增主流程，不替代 `why/how/task`，只在执行期防止猜、扩写、乱改、静默失败。

适用：所有改动型任务。Quick Fix 只执行最小闭包；标准开发按触发条件展开。

---

## 1) 判断前置：先澄清风险，不默认猜

- 不确定点必须显式化；影响架构、数据、权限、安全、验收或用户可见行为时，先问或写入待确认假设。
- 小任务不因普通细节不确定而停滞；能通过只读取证确认的，先取证再判断。
- 若存在多个合理解释，优先列出 2–3 个解释并选择当前采用的一个；选择依据必须来自 `[SRC:USER]`、`[SRC:CODE]` 或 `[SRC:TOOL]`。
- 当发现更简单方案能满足成功标准时，应主动收敛，不为“完整感”扩展范围。

---

## 2) 最小改动：默认 surgical_edit

- `surgical_edit`：只改成功标准要求的最小 Workset；默认不新增文件、不顺手重构、不格式化无关代码。
- 只能清理自己引入的临时代码、重复实现、错误修改；不能借任务名义“顺手改善”相邻模块。
- 新增抽象、跨层改动、公共契约变化、重构预算，必须先写入执行域声明并说明必要性。
- 代码风格以当前代码库为准；即使不同意现有风格，也不得暗中分叉一套新约定。

---

## 3) 写前先读：读最小必要上下文

首次写入前，至少确认：
- 将改文件的当前实现。
- 直接调用方 / 导出入口 / 共享工具中与本次改动相关的最小片段。
- 是否已有相似实现、类型、错误码、配置项或测试入口可复用。

通用取证顺序：
- 若运行时暴露原生搜索/读取工具，先用 `fastgrep.*`、`workspace.read_context/read_slice` 等 native tools 做定位、相似实现检索和最小片段读取。
- shell `rg` / `Get-Content` / `cat` 只在原生工具不可用、失败、或无法表达查询/读取时作为 fallback only；必须记录降级原因。
- 同一查询或同一读取范围不得在原生工具成功后再用 shell 重复跑；确需复查时先写明 fallback/验证理由。

Delphi/Pascal 任务的额外顺序：
- 仅在 Delphi/Pascal 项目或任务中触发；若 Delphi 语义工具可用，先查 `delphi/getIndexStatus`；`missing` 用 `delphi/indexWorkspace`，`stale` 用 `delphi/refreshIndex`，失败或不可用时记录降级原因；证据门禁见 `references/delphi-evidence-gate.md`。
- 写前证据优先级为 `delphi/getSymbolsOverview` -> `delphi/findDefinition` -> `delphi/findReferences` -> `delphi/impactAnalysis` -> 少量精读；`rg` 只作兜底或补充确认。
- `partial`、`warnings`、`risks` 必须显式记录；索引状态、overview 和 hooks 提醒都不是真值，不能替代代码事实、方案包状态或验证证据。
- 只有真实 `delphi.*` tool call 或 `item/tool/call namespace=delphi` 证据，才能声明已使用 Delphi 语义工具；`dynamicTools` 注入、工具列表可见、索引 ready、代码里有 executor 都只是准备证据。
- 如果实际 tool calls 只有 `rg` / `Get-Content` / shell 读取，必须明确写“文本搜索 fallback”，不得把 fallback 说成语义查询。

停止条件：
- 已能解释“为什么改这里”。
- 已能说明“为什么不需要改相邻模块”。
- 已能给出本轮 `verify_min` 或说明验证入口缺失。

---

## 4) 冲突显式化：不平均两种模式

当代码库存在两套互相冲突的模式时：
- 不写“两边都不像”的折中实现。
- 优先选择：更近修改、更有测试覆盖、更主流调用路径、更符合项目边界的一边。
- 另一种模式只记录为待清理/待统一，不在本任务里顺手重构。
- 选择原因写入 `task.md##上下文快照` 的“决策”区；若影响架构/契约，先请求用户确认。

---

## 5) 确定性交给脚本：模型只做判断

优先交给脚本、hook 或校验器处理：
- 路由状态、恢复状态、归档状态。
- 重试、去重、锁、结构字段检查。
- hook payload 解析、`_current.md` 指针合法性、方案包结构校验。
- deterministic transform（格式转换、批量替换、清单比对）。

模型负责：
- 意图建模、风险判断、方案取舍、冲突解释、验证证据解读。

规则：如果代码/脚本能稳定回答，就不要让模型凭感觉重复判断；主流程只消费结构化结果。

---

## 6) 验证意图：没有证据就不能声称通过

- `verify_min` 必须映射成功标准；测试应验证“为什么这件事重要”，不只是表面返回值。
- 没有真实命令输出、测试输出或可复现手动验证记录时，不得声称“验证通过”。
- 验证失败、跳过验证、只做静态审查，都必须显式写出原因和下一步可执行动作。
- 发现验证与成功标准不匹配时，先补验证目标或调整成功标准，不用无关测试冒充覆盖。

---

## 7) 检查点：只在显著状态变化时落盘

不要求每一步都写快照；命中以下任一情况必须写 `task.md##上下文快照`：
- 关键决策、需求/约束变化、执行域变化。
- 阻断失败、隐性空转、同一 Workset 反复修改。
- 上下文预算 Yellow/Red、压缩前后、续作/模型断层。
- 完成阶段性任务且下一步会跨文件/跨阶段/跨会话。

检查点只写续作必需信息：Workset、repo_state、当前决策、验证证据或失败证据、下一步唯一动作。

---

## 8) Fail loud：失败和跳过必须可见

- “完成”不能包含静默跳过项。
- “测试通过”不能包含未执行或失败的测试。
- 工具失败、脚本失败、hook SKIP、验证未覆盖成功标准，都必须显式暴露。
- 若继续执行会掩盖不确定性，应停止并转入失败协议或断层恢复协议。
